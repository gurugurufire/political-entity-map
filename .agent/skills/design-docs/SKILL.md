# 📝 Design Docs (設計・ドキュメント作成)

## 概要
開発ライフサイクルの各段階（研究、計画、仕様確定）において、標準化された高品質なドキュメントを生成します。

## ワークフロー

### 1. ドキュメント種類の判定
ユーザーの指示または現在の作業フェーズから、以下のいずれを作成すべきか判断します。

| フェーズ          | 文書種別   | 格納先           | 作成タイミング                             |
| :---------------- | :--------- | :--------------- | :----------------------------------------- |
| **Research**      | 研究・構想 | `docs/research/` | 新機能の検討、未知の技術の調査時。         |
| **Planning**      | 計画・設計 | `.agent/plans/`  | **実装着手直前**。具体的な手順を固める時。 |
| **Specification** | 仕様確定   | `docs/specs/`    | 実装完了・安定後。システムの正典とする時。 |

### 2. テンプレートの適用
`assets/templates/` 配下にある対応するテンプレートを `view_file` で読み込み、内容を構築します。

### 3. 書き出しと記録
- 指定されたディレクトリにファイルを書き出します。
- `AgentHistory.md` および `TaskBoard.md` を更新し、設計が完了したことを記録します。

## 記述標準 (HD要件)

### 📊 Mermaid図の厳格運用
- 複雑なロジックは必ず図解する。
- **引用符ルール**: ノード内のテキストやラベルは、必ず二重引用符 `""` で囲むこと（構文エラー防止）。
    - ✅ `Node["日本語ラベル"]`

### 🧠 意思決定（Why）の明文化
- 単に「結果」を書くのではなく、「なぜその手法を選んだか（Rationale）」を必ず記述する。
- 採用しなかった代替案についても簡潔に触れる。

### 📍 定量的・具体的な記述
- 絶対パス、DB ID、変数名、APIエンドポイントなど、曖昧さを排除した具体的なパラメータを明記する。
